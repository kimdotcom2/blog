## 컨테이너 기술과 Docker를 명확히 구분하여 설명하세요. 컨테이너 기술이 Docker 이전에도 존재했던 개념임을 언급하고, Docker가 컨테이너 기술을 구현한 하나의 도구라는 관점에서 설명해주세요. 또한, Docker 외에 컨테이너 기술을 구현한 다른 도구의 예시를 들어보세요.

컨테이너는 운영체제 수준의 가상화 기술을 이용해 프로세스와 실행 환경을 격리하는 개념이다.
"어플리케이션 + 필요한 실행 환경"을 하나의 패키지로 묶어 어디서나 동일하게 실행되도록 만드는 기술이다.

Docker 이전에는 다음과 같은 기술이 존재했다.

- LXC(Linux Containers)
- Solaris Zones
- FreeBSD Jails

Docker는 이러한 컨테이너 기술을 보다 쉽게 사용할 수 있도록 만든 도구이다. 
Docker는 컨테이너 이미지를 생성, 배포, 관리하는 데 필요한 다양한 기능을 제공하며, 개발자들이 컨테이너를 손쉽게 활용할 수 있도록 돕는다.

Docker의 특징은 다음과 같다.

- 이미지 기반 배포 : Docker 이미지는 어플리케이션과 그 실행 환경을 포함한 패키지로, 어디서나 동일하게 실행 가능하다.
- 경량화 : Docker 컨테이너는 호스트 운영체제의 커널을 공유하므로, 가상 머신에 비해 훨씬 가볍고 빠르다.
- 이식성 : Docker 이미지는 다양한 환경에서 동일하게 실행될 수 있어, 개발, 테스트, 프로덕션 환경 간의 차이를 최소화한다.
- 생태계 : Docker Hub와 같은 중앙 저장소를 통해 다양한 이미지를 공유하고 활용할 수 있다.
- 컨테이너 정의 : Dockerfile을 통해 컨테이너 이미지를 코드로 정의하고 버전 관리할 수 있다.

## 컨테이너 오케스트레이션의 개념과 필요성을 설명하고, Docker 단독 사용 환경과 비교하여 컨테이너 오케스트레이션이 해결하는 주요 문제점 3가지(자동 확장, 자가 복구, 선언적 인프라)를 설명하세요.

컨테이너 오케스트레이션은 다수의 컨테이너를 자동으로 배포, 관리, 확장하는 프로세스를 의미한다.

Docker를 단독 사용하면 다음과 같은 문제점이 발생할 수 있다.

1. 수동 확장 : 트래픽 증가에 따라 컨테이너를 수동으로 추가하거나 제거해야 하므로, 운영 부담이 크고 신속한 대응이 어렵다.
2. 장애 복구 : 컨테이너가 실패했을 때 자동으로 재시작하거나 대체 컨테이너를 생성하는 기능이 부족하다.
3. 선언적 인프라 관리 : 인프라 상태를 코드로 정의하고 관리하는 것이 어려워, 일관된 환경 유지가 어렵다.

컨테이너 오케스트레이션 도구는 이러한 문제점을 해결한다.

1. 자동 확장 : 오케스트레이션 도구는 트래픽 변화에 따라 컨테이너 수를 자동으로 조절하여, 서비스의 가용성과 성능을 유지한다.
2. 자가 복구 : 컨테이너가 실패하면 자동으로 재시작하거나 새로운 컨테이너를 생성하여, 서비스의 안정성을 높인다.
3. 선언적 인프라 : 인프라 상태를 코드로 정의하고 관리할 수 있어, 일관된 환경을 유지하고 변경 사항을 추적할 수 있다.

그 외에도 로드 밸런싱, 서비스 디스커버리, 롤링 업데이트 등의 기능을 제공하여, 대규모 컨테이너 환경을 효율적으로 관리할 수 있다.

대표적인 컨테이너 오케스트레이션 도구로는 Kubernetes, Docker Swarm, Apache Mesos 등이 있다.
